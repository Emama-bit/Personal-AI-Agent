一、版本调整说明

1.1 从“AI 替身市场”调整为“AI Skill 试单市场”

初版方案的核心是“把人的经验训练成可交易的 AI 替身”。第二版在此基础上收敛为更适合 MVP 落地的双边平台：

- 需求侧：不需要先懂自由职业者、不需要先登录，直接浏览 Skill 市场，输入模糊需求，由平台推荐可试用的 Skill AI。
- 响应侧：自由职业者、顾问、创作者、技术人员等可以发布自己的 Skill AI，用经验、流程、案例和服务边界生成可试用入口。
- 平台侧：提供 Skill 展示、AI 试用、需求撮合、线索管理、风险提示和交易辅助，不直接承诺最终服务结果。

第二版不再把“支付/托管”作为当前最优先主线，而是先补齐主要功能：

1. 需求侧找到合适 Skill 的能力。
2. 响应侧发布和管理 Skill 的能力。
3. 双方从 AI 试用过渡到真人服务线索的能力。
4. 平台风险提示、去责任化说明和线索状态管理能力。

支付、托管、抽佣作为后续商业闭环保留，但当前只做轻量模拟，不作为核心开发阻塞项。

1.2 当前项目已完成的 MVP 基础

当前代码已完成以下原型模块：

- 首页：AI Skill 试单市场、需求侧/响应侧入口、风险提示。
- 需求雷达：需求方输入一句模糊需求，平台即时推荐匹配 Skill，并展示匹配理由。
- Skill 市场：搜索、领域筛选、排序、Skill 剧场卡片。
- Skill 详情页：AI 试用聊天、评分、风险提示、真人服务意向提交。
- 登录策略：需求侧浏览市场不强制登录；试用 Skill AI 时弹出轻量登记；响应侧进入工作台。
- 发布 Skill：响应侧填写 Skill 名称、类目、能力介绍、适用场景、真人服务说明、风险边界，并上传素材。
- Dashboard：角色模拟，支持需求侧个人中心和响应侧工作台。
- 线索 CRM：新线索、已联系、方案沟通、已成交/关闭。
- 线索解锁：联系方式默认遮罩，响应侧模拟支付解锁后可见。
- 报价单与托管订单：已有轻量模拟，但后续优先级低于主要撮合功能。

二、项目定位

2.1 一句话定位

一个让需求方先用 AI Skill 低成本试单，再决定是否连接真人自由职业者的双边平台。

2.2 核心价值

对需求侧：

- 不用先判断“该找谁”，而是先描述需求。
- 不用一开始就付高价咨询费，先通过 Skill AI 看能力、方法和风格。
- 不用完全相信简历或服务介绍，可通过试用结果判断是否继续联系真人。

对响应侧：

- 不再只靠简历、作品集和私聊获客。
- 可以把个人方法、流程、案例、服务边界沉淀成可试用的 Skill AI。
- 低价值重复咨询由 AI 承接，高价值需求转为真人服务线索。

对平台侧：

- 平台提供展示、试用、撮合、线索和交易辅助。
- 平台不承诺 AI 输出构成法律、医疗、金融、投资等专业最终意见。
- 平台不直接担保真人服务结果，具体服务范围、报价、交付和责任边界由双方确认。

三、用户与场景

3.1 需求侧用户

1. 小微企业主
   - 需求：内容、商业、运营、技术等具体问题需要低成本判断。
   - 痛点：直接找顾问贵，找自由职业者质量不稳定，通用 AI 又缺少领域经验。

2. 职场新人 / 转岗人群
   - 需求：简历、面试、职业路径、项目表达。
   - 痛点：不知道自己问题在哪，也不知道该找什么类型的人帮忙。

3. 创业者 / 副业探索者
   - 需求：BP、商业模式、获客路径、MVP 技术方案。
   - 痛点：早期不确定性高，不适合一开始就购买完整咨询。

4. 个人消费者
   - 需求：具体小问题需要专业判断。
   - 痛点：事情太小，不想正式请专家；但通用 AI 又不够可信。

3.2 响应侧用户

1. 自由职业者
   - 目标：获得更多高质量线索，减少重复沟通。
   - 平台价值：AI 先筛选需求，真人只承接更明确的高价值需求。

2. 资深从业者
   - 目标：把经验规模化变现。
   - 平台价值：将方法论、案例和判断流程转成可试用 Skill。

3. 内容创作者 / 顾问 / 技术人员
   - 目标：让用户先体验能力，再转化为服务。
   - 平台价值：Skill AI 成为可交互的作品集。

四、MVP 产品结构

4.1 需求侧流程

入口一：自由浏览 Skill 市场

1. 进入首页。
2. 查看需求侧/响应侧说明和风险提示。
3. 浏览 Skill 卡片。
4. 使用搜索、分类、排序筛选 Skill。
5. 进入 Skill 详情页。
6. 如要真正试用 Skill AI，则弹出轻量登记。
7. 试用后，如需求较高价值，可提交真人服务意向。

入口二：需求雷达匹配

1. 需求方输入一句模糊需求。
2. 平台根据领域关键词、Skill 描述、素材量和评分进行轻量匹配。
3. 返回推荐 Skill、匹配分和匹配理由。
4. 用户可直接进入 Skill 试镜，或查看同类 Skill。

入口三：需求侧个人中心模拟

1. 查看自己提交过的服务意向。
2. 查看响应侧报价单。
3. 查看是否接受托管交易。
4. 后续可扩展为需求发布广场、报价对比和服务记录。

4.2 响应侧流程

1. 进入响应侧登录页。
2. 发布 Skill AI：
   - Skill 名称。
   - 技能类目。
   - 能力介绍。
   - 适用场景。
   - 真人服务说明。
   - 风险边界。
   - 上传 Skill 素材。
3. 在 Dashboard 管理 Skill：
   - 查看已发布 Skill。
   - 查看上架/暂停状态。
   - 查看素材、流程、边界、转化等 Skill DNA 指标。
4. 管理线索：
   - 新线索。
   - 已联系。
   - 方案沟通。
   - 已成交/关闭。
5. 解锁联系方式：
   - 默认遮罩需求方联系方式。
   - 模拟支付线索解锁费后可见。
6. 生成报价单：
   - 平台根据需求长度和服务范围生成模拟报价。
   - 后续可演化为人工填写报价、套餐报价和议价流程。

4.3 平台侧能力

- 市场展示。
- 需求匹配。
- Skill AI 试用。
- 线索生成。
- 线索状态管理。
- 风险提示。
- 去责任化声明。
- 评分系统。
- 报价与托管模拟。

五、当前技术架构

5.1 技术栈

- 前端与后端：Next.js App Router。
- UI：React Client Components + 全局 CSS。
- 数据存储：本地 JSON 文件模拟数据库。
- AI 调用：已预留 AI 适配层，支持 mock/真实模型切换。
- RAG：本地文本分块与轻量检索。

5.2 当前数据文件

- data/proxies.json：Skill AI 列表。
- data/chunks.json：Skill 素材分块。
- data/ratings.json：评分数据。
- data/leads.json：真人服务线索。
- data/quotes.json：报价单。
- data/orders.json：托管订单模拟。

5.3 当前核心路由

页面路由：

- /：首页、需求雷达、Skill 市场。
- /create：响应侧发布 Skill。
- /proxy/[id]：Skill 详情、AI 试用、评分、真人服务意向。
- /proxy/[id]/edit：编辑 Skill。
- /dashboard：双角色个人中心。
- /leads：完整线索后台。
- /login：响应侧登录入口，需求侧不做完整登录页。

API 路由：

- GET/POST /api/proxy：Skill 列表与创建。
- GET/PATCH/DELETE /api/proxy/[id]：Skill 详情、更新、删除。
- POST /api/proxy/[id]/chat：Skill AI 对话。
- GET/POST /api/proxy/[id]/rate：评分。
- GET/POST /api/leads：线索列表与创建。
- PATCH /api/leads/[id]：线索状态更新、联系方式解锁。
- GET/POST /api/quotes：报价单列表与创建。
- PATCH /api/quotes/[id]：报价接受。
- GET/POST /api/orders：托管订单列表与创建。
- PATCH /api/orders/[id]：订单状态更新。
- POST /api/upload：上传文本素材。

六、第二版功能任务拆解

6.1 P0：主要功能优先

任务 1：需求雷达继续增强

目标：
让需求方不用理解类目，也能快速找到合适 Skill。

已完成：
- 首页新增需求雷达。
- 支持示例需求。
- 支持匹配分、匹配理由、进入试镜、看同类 Skill。

下一步：
- 增加“需求拆解结果”：把用户输入拆成目标、场景、风险、建议先试的 Skill 类型。
- 增加“推荐组合路径”：例如商业 Skill + 内容 Skill + 技术 Skill。
- 增加“不确定时推荐发布需求”的入口。

任务 2：需求发布广场

目标：
需求方不仅能找 Skill，也能公开发布需求，让响应侧主动认领。

功能：
- 需求方填写需求标题、预算范围、期望交付、行业场景、是否接受 AI 先试单。
- 需求展示在需求广场。
- 响应侧可点击“我能承接”。
- 平台生成响应侧候选列表。

MVP 实现：
- 使用本地 JSON 存储 demands。
- 页面：/demands。
- API：/api/demands。
- Dashboard 需求侧展示“我发布的需求”。
- Dashboard 响应侧展示“可认领需求”。

任务 3：Skill 对比功能

目标：
需求方在多个 Skill 之间做选择。

功能：
- 市场卡片支持加入对比。
- 对比维度：
  - 类目。
  - 能力说明。
  - 适用场景。
  - 素材数量。
  - 评分。
  - 风险边界。
  - 是否可转真人服务。
- 对比后可进入任一 Skill 试用。

任务 4：Skill 详情页增强

目标：
让 Skill 不只是聊天页，而像一个“可试用的自由职业者服务页”。

功能：
- 展示适用场景。
- 展示真人服务说明。
- 展示风险边界。
- 展示用户可问的示例问题。
- 展示“是否适合我”的引导问题。

任务 5：响应侧 Skill 管理增强

目标：
让响应侧知道自己的 Skill 表现如何。

功能：
- Skill 状态：上架、暂停、草稿。
- Skill 完整度。
- 线索转化数。
- 被试用次数。
- 被收藏/加入对比次数。
- 建议补充哪些素材。

6.2 P1：线索和服务承接

任务 6：线索 CRM 继续增强

已完成：
- 新线索、已联系、方案沟通、已成交/关闭。
- 联系方式解锁。

下一步：
- 每条线索增加备注。
- 增加下一步跟进时间。
- 增加线索质量评分。
- 增加“无效线索原因”。

任务 7：报价单半结构化

当前已有模拟报价。

下一步：
- 响应侧可手动编辑报价金额。
- 可选择服务包模板。
- 可添加交付物列表。
- 可添加不包含内容。
- 可添加交付时间。

6.3 P2：支付和托管辅助

当前原则：
支付环节不是当前主线，暂不阻塞主要功能。

保留能力：
- 报价单。
- 需求侧接受托管。
- 平台抽佣展示。
- 托管订单状态模拟。

后续接入条件：
- 有真实用户。
- 有真实线索转化。
- 需求侧愿意在平台内完成服务。
- 需要评价、争议、履约记录沉淀时再强化。

七、风险与责任边界

7.1 平台声明

平台仅提供：

- Skill AI 展示。
- Skill AI 试用。
- 供需撮合。
- 线索记录。
- 报价和交易辅助。

平台不承诺：

- AI 输出一定正确。
- 真人服务一定达到预期。
- 法律、医疗、金融、投资等专业最终意见。
- 双方线下沟通和线下交易的履约结果。

7.2 响应侧声明

响应侧发布 Skill 时必须确认：

- 不上传无权使用的资料。
- 不冒充他人资历。
- 明确服务边界。
- 明确不承诺的结果。
- 真人服务前需与需求侧另行确认范围、报价、交付和责任。

7.3 需求侧声明

需求侧使用 Skill 时必须确认：

- AI 结果仅供试用和初步判断。
- 是否联系真人由需求侧自行决定。
- 是否提供联系方式和需求材料由需求侧自行决定。
- 重要决策需自行寻求合格专业人士确认。

八、阶段目标与验收标准

8.1 当前 MVP 阶段目标

目标不是先做复杂支付，而是验证：

1. 需求方是否愿意用一句需求来找 Skill。
2. 需求方是否愿意先试用 Skill AI。
3. 试用后是否愿意提交真人服务意向。
4. 响应侧是否愿意发布自己的 Skill AI。
5. 响应侧是否认为线索有价值。

8.2 当前阶段验收标准

需求侧：

- 可以不登录浏览市场。
- 可以输入需求并获得推荐 Skill。
- 可以进入 Skill 详情页试用。
- 可以提交真人服务意向。
- 可以在个人中心看到提交记录和报价。

响应侧：

- 可以发布 Skill。
- 可以管理 Skill。
- 可以查看线索。
- 可以推进线索状态。
- 可以解锁联系方式。
- 可以生成报价单。

平台侧：

- 所有关键路径都有风险提示。
- 需求侧和响应侧职责边界清晰。
- 支付和托管只是模拟，不影响核心流程。

十、第二版结论

第二版项目不再把“AI 替身”理解成单纯的简历或虚拟人，而是定义为：

自由职业者把自己的经验、流程、案例和服务边界沉淀成可试用的 AI Skill；需求侧先用 Skill AI 判断能力和风格，再决定是否进入真人服务。

当前最重要的不是支付，而是让平台形成三件事：

1. 需求可以被理解。
2. Skill 可以被试用。
3. 真人服务线索可以被管理。

只要这三件事跑通，后续的线索解锁、报价、托管、抽佣、订阅和企业采购才有真实基础。
